# Ground Station Repository Analysis Report

## Executive Summary

The ground-station repository (github.com/sgoudelis/ground-station) is a comprehensive, full-featured open-source software solution for satellite tracking and radio communication. Built with Python/FastAPI backend and React frontend, it provides real-time satellite tracking, multi-target observation scheduling, SDR integration, signal demodulation/decoding, and an sophisticated automated observation system. This report details the repository structure, key features, architectural patterns, and identifies components that could be integrated into SkyCore.

## 2. Main Features and Capabilities

### 2.1 Real-time Satellite Tracking

The system provides comprehensive satellite tracking capabilities with high-precision orbital models. Key aspects include:

- **TLE Data Management**: Automatic updates from CelesTrak and SatNOGS databases
- **Multi-target Tracking**: Simultaneous tracking of multiple satellites with independent hardware control paths
- **Orbital Propagation**: Support for both TLE and OMM (Orbital Mean-element Message) formats
- **Central Body Abstraction**: Extensible design supporting Earth, Moon, and Mars orbit propagation

### 2.2 Hardware Integration

The repository demonstrates extensive hardware integration patterns:

- **SDR Device Support**:
  - RTL-SDR (USB and tcp modes)
  - SoapySDR devices (local and remote via SoapyRemote)
  - UHD/USRP radios
  - SigMF Playback (virtual SDR for recording playback)

- **Antenna Rotator Control**: Hamlib-compatible rotators with automatic satellite tracking
- **Rig Control**: CAT-controlled radios with Doppler correction support

### 2.3 Signal Processing Pipeline

A sophisticated multi-stage signal processing architecture:

1. **IQ Acquisition**: Raw sample streaming from SDR devices
2. **IQ Broadcaster**: Pub/sub pattern for distributing samples to multiple consumers
3. **FFT Processor**: Real-time spectrum computation and waterfall generation
4. **Demodulators**: FM, SSB, AM, FM-Stereo support with internal and normal modes
5. **Decoders**: SSTV, AFSK, GFSK, GMSK, BPSK, LoRa, Morse code
6. **Audio Broadcaster**: Distribution of demodulated audio to decoders and UI

### 2.4 Automated Observation System

A sophisticated scheduled observation system that automatically executes satellite passes:

- **Monitored Satellites**: Templates with hardware configs, signal parameters, and task definitions
- **Pass Scheduling**: APScheduler-based scheduling at AOS (Acquisition of Signal) time
- **Task Composition**: Multiple concurrent tasks (IQ recording, audio recording, decoding, transcription)
- **Hardware Orchestration**: Automatic SDR, rotator, and rig control during observations
- **Status Management**: Tracking of scheduled, running, completed, failed, cancelled, and missed states

### 2.5 Data Decoding

Multiple protocol decoders are implemented:

- **SSTV Decoder**: Robot36, PD120, etc. (fully working)
- **AFSK Decoder**: For packet radio (in development)
- **GMSK/FSK/BPSK Decoders**: Various digital modes
- **LoRa Decoder**: Experimental support
- **GNSS-SDR Decoder**: For GPS/GLONASS signals
- **Morse Code Decoder**: CW demodulation

### 2.6 IQ Recording and Playback

Using SigMF (Signal Metadata Format):

- Automatic metadata capture (frequency, sample rate, timestamp)
- Satellite tagging of recordings
- Waterfall snapshot preservation
- Virtual SDR playback device for analysis

### 2.7 AI-Powered Transcription

Integration with external AI services:

- Gemini Live or Deepgram for real-time speech-to-text
- Privacy-conscious, user-keyed operation
- Optional translation support
- File output to transcriptions directory

### 2.8 Web Interface

Modern React-based UI with:

- Material-UI components
- Real-time updates via Socket.IO
- Responsive design for desktop, tablet, and mobile
- Interactive satellite maps (Leaflet)
- Spectrum/waterfall displays
- Multi-language support (i18n)

## 3. Key Architectural Patterns

### 3.1 Backend Architecture (FastAPI + Socket.IO)

**Core Components**:

- **app.py**: Main entry point with uvicorn server startup
- **server/startup.py**: Application initialization, route configuration
- **handlers/socket.py**: Socket.IO event routing and handler registration
- **pipeline/orchestration/processmanager.py**: Central orchestrator for SDR workers

**Pattern: Entity-Handler Registry**
The system uses a handler registry pattern where each entity module registers its handlers:

```python
# handlers/routing.py
handler_registry = {}

def dispatch_request(sio, cmd, data, logger, sid, registry):
    # Routes commands to registered handlers
```

Each entity (satellites, hardware, tracking, etc.) has its own handler module that registers callbacks.

### 3.2 Database Design (SQLAlchemy + SQLite/PostgreSQL)

**Models** (`db/models.py`):
- `Satellites`: Satellite definitions with TLE data
- `SatelliteOrbits`: Separate orbit table for multi-body support
- `SatelliteGroups`: Categorization of satellites
- `SDRs`: SDR device configurations
- `Rotators`: Antenna rotator configurations
- `Rigs`: Radio/rig configurations
- `Locations`: Ground station location data
- `ScheduledObservations`: Observation scheduling
- `DecodedMessages`: Decoded data storage

**Key Patterns**:
- Custom TypeDecorator classes (`AwareDateTime`, `JsonField`) for timezone handling and JSON serialization
- Alembic migrations for schema version control
- Async session support for non-blocking operations

### 3.3 Pub/Sub Broadcasting Pattern

**IQ Broadcaster** (`pipeline/streaming/iqbroadcaster.py`):
- Producers don't know about consumers
- Independent queues with deep-copied samples per subscriber
- Slow consumers: messages dropped rather than blocking producer
- Statistics tracking: delivered/dropped message counts

**Audio Broadcaster**:
- Distributes demodulated audio to decoder and UI subscribers
- Used for internal demodulators feeding decoders

### 3.4 Process Management

**ProcessManager** orchestrates multiple specialized managers:
- `DemodulatorManager`: Manages demodulator consumers
- `RecorderManager`: Manages IQ/Audio recording
- `DecoderManager`: Manages decoder consumers
- `TranscriptionManager`: Per-VFO transcription
- `ProcessLifecycleManager`: Process start/stop/configure

### 3.5 Session Management

**Session Service** (`session/service.py`):
- Isolated VFO sessions for different contexts
- Namespace isolation: `"internal:<observation_id>"` for automated observations
- User sessions operate independently with shared SDR capability

### 3.6 Observation Execution

**ObservationExecutor** (`observations/executor.py`):
- Triggered by APScheduler at AOS time
- Orchestrates SDR, VFO, decoder, and tracker components
- Task handlers: `DecoderHandler`, `RecorderHandler`, `TranscriptionHandler`, `TrackerHandler`

### 3.7 Tracker Architecture

**Multi-component tracking system**:
- `TrackerManager`: Core tracker state machine
- `TrackerRunner`: Executes tracker instances
- `TrackerLogic`: Core tracking calculations
- `RigHandler`: Doppler correction for rigs
- `RotatorHandler`: Antenna pointing control
- IPC communication via shared memory

### 3.8 Frontend Architecture (React + Redux Toolkit)

**Component Structure**:
- Smart/dumb component pattern
- Custom hooks for complex logic
- Redux Toolkit for state management
- Socket.IO client for real-time updates

**Key Directories**:
- `components/`: Feature-based component organization
- `hooks/`: Reusable React hooks
- `config/`: Application configuration
- `i18n/`: Internationalization with locale files

## 4. Specific Modules and Functionality

### 4.1 Orbits Module (`backend/orbits/`)

**Purpose**: Planet-aware orbit service abstractions

**Key Classes**:
- `OrbitPropagator`: Abstract base for propagation adapters
- `EarthTlePropagator`: Native Earth/TLE propagation
- `EarthOmmCompatibilityPropagator`: OMM to TLE conversion
- `OrbitService`: Central service for loading orbit state

**Design**: Strategy pattern for different central body/orbit model combinations

### 4.2 Observations Module (`backend/observations/`)

**Purpose**: Automated satellite pass observation system

**Key Files**:
- `executor.py`: Observation lifecycle management
- `generator.py`: Pass prediction and observation generation
- `sync.py`: TLE synchronization
- `tasks/`: Task handlers for decoders, recorders, transcription, tracking

**Features**:
- APScheduler integration for timing
- Conflict detection and resolution
- Status event emission via Socket.IO

### 4.3 Demodulators Module (`backend/demodulators/`)

**Implemented Demodulators**:
- `fmdemodulator.py`: FM demodulation (53KB)
- `ssbdemodulator.py`: Single Sideband
- `amdemodulator.py`: AM demodulation
- `fmstereodemodulator.py`: FM Stereo

**Decoder Base Classes**:
- `basedecoder.py`: Common decoder functionality
- `basedecoderprocess.py`: Process-based decoder execution

### 4.4 Tracker Module (`backend/tracker/`)

**Purpose**: Real-time satellite tracking with hardware control

**Key Components**:
- `manager.py`: Tracker state machine management (35KB)
- `logic.py`: Core tracking calculations (32KB)
- `righandler.py`: Radio control with Doppler correction (33KB)
- `rotatorhandler.py`: Antenna rotator control (23KB)
- `runner.py`: Tracker instance execution
- `state.py`: Tracker state definitions

### 4.5 Hardware Probing (`backend/hardware/`)

**SDR Enumeration**:
- `rtlsdrenum.py`: RTL-SDR device probing
- `soapysdrlocalprobe.py`: Local SoapySDR devices
- `soapysdrremoteprobe.py`: Remote SoapySDR via network
- `uhdprobe.py`: USRP/UHD devices
- `sigmfprobe.py`: SigMF recording probe

### 4.6 VFO Management (`backend/vfos/`)

**Purpose**: Virtual Frequency Oscillator management for multi-channel operation

**Features**:
- Per-VFO configuration (frequency, mode, bandwidth)
- Internal VFO numbering for automated observations
- State management for VFO updates

### 4.7 FFT Processing (`backend/fft/`)

**Components**:
- `processor.py`: FFT computation
- `waterfallgenerator.py`: Waterfall display data
- `averager.py`: Signal averaging

### 4.8 Pipeline Managers (`backend/pipeline/managers/`)

**Consumer Managers**:
- `demodulatormanager.py`: Demodulator lifecycle
- `decodermanager.py`: Decoder process management (43KB)
- `recordermanager.py`: IQ/Audio recording
- `audiorecordermanager.py`: Audio-only recording
- `transcriptionmanager.py`: AI transcription (20KB)
- `consumerbase.py`: Common consumer functionality

### 5.4 Specific Code Patterns

#### Type-Safe Configuration
```python
# backend/orbits/service.py
@dataclass(frozen=True)
class OrbitState:
    object_id: int
    object_name: Optional[str]
    central_body: CentralBody
    model_kind: OrbitModelKind
```

#### Handler Registry Pattern
```python
# backend/handlers/routing.py
handler_registry = {}

def register_handler(entity: str, command: str, handler):
    handler_registry[(entity, command)] = handler
```

#### Manager Delegation Pattern
```python
# backend/pipeline/orchestration/processmanager.py
class ProcessManager:
    def __init__(self, sio=None, event_loop=None):
        self.demodulator_manager = DemodulatorManager(self.processes)
        self.recorder_manager = RecorderManager(self.processes, sio=self.sio)
        self.decoder_manager = DecoderManager(self.processes, self.demodulator_manager)
```

## 6. Summary

The ground-station repository represents a mature, well-architected application for satellite ground station operations. Its key strengths include:

1. **Comprehensive Feature Set**: Satellite tracking, multi-target support, automated observations, SDR integration, signal processing, and data decoding

2. **Clean Architecture**: Clear separation between backend (Python/FastAPI), frontend (React), and worker processes with well-defined interfaces

3. **Extensible Design**: Strategy patterns for orbit propagation, plugin-like handler registration, and modular pipeline components

4. **Production-Ready**: Docker deployment, database migrations, performance monitoring, comprehensive error handling

5. **Real-time Capabilities**: Socket.IO for live updates, pub/sub for signal distribution, async operations throughout

For SkyCore integration, the most valuable contributions would be:
- The automated observation scheduling system
- The orbital propagation abstraction layer
- The hardware abstraction patterns for SDR/radio/rotator control
- The pub/sub signal distribution architecture
- The well-designed database schema for satellite and mission data
